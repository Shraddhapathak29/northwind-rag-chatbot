# Northwind Gadgets — Dual-Mode Agentic RAG Chatbot

A production-style GenAI chatbot that answers questions about a fictional company
("Northwind Gadgets") using **two distinct retrieval strategies inside one agent**,
and **decides per question which to use**:

1. **Agentic RAG** over policy documents (vector retrieval + citations).
2. **Text-to-SQL** over a structured `orders` table (generated, validated, executed).

The agent routes each question to the right tool — or **both** for mixed questions —
and returns a safe fallback (`"I don't have that information."`) for out-of-scope queries.

> Built for the EMB **AI Engineer** technical assessment.


## 1. Architecture

```
┌──────────────┐   POST /chat/stream (SSE, token-level)   ┌─────────────────────────────────┐
│  Next.js UI  │ ───────────────────────────────────────► │  FastAPI                        │
│  (chat)      │                                           │   ┌───────────────────────────┐ │
│  shows:      │ ◄─────────────────────────────────────── │   │ Agent (JSON routing)       │ │
│  • tool used │   events: tools│citations│sql│token│done  │   │  routes per question       │ │
│  • citations │                                           │   └────┬──────────────┬────────┘ │
│  • SQL+rows  │                                           │        ▼              ▼          │
└──────────────┘                                           │  search_documents  query_orders  │
                                                           │   (RAG, pgvector)  (text-to-SQL) │
                                                           │        │              │          │
                                                           │        ▼              ▼          │
                                                           │   ┌──────────────────────────┐   │
                                                           │   │ PostgreSQL + pgvector     │   │
                                                           │   │ • documents(chunk,vector) │   │
                                                           │   │ • orders (200 rows)       │   │
                                                           │   └──────────────────────────┘   │
                                                           └─────────────────────────────────┘
```

### Request lifecycle (per question)
1. **Phase A — routing (non-streamed):** the LLM returns a small **JSON decision**
   — `{search_query, orders_question, direct_answer}` — saying which tool(s) to use
   (or a direct answer for greetings / out-of-scope). We parse it ourselves and run the
   chosen tools, capturing the **citations** (RAG) and the **generated SQL + rows**.
   For mixed questions it sets *both* `search_query` and `orders_question`.
2. **Phase B — synthesis (streamed):** a plain completion writes the final, grounded
   answer from the tool results, **token by token**, streamed to the browser over SSE.
3. If **no tool** is needed (greeting / out-of-scope), the JSON `direct_answer` — the safe
   fallback for out-of-scope — is streamed directly.

> **Why JSON routing instead of native function-calling?** Open models served via the
> Hugging Face router emit native tool calls inconsistently, and some providers reject them
> with `tool_use_failed`. Asking for a tiny strict-JSON decision (plain text in, plain text
> out) is reliable across every provider, and parsing it ourselves keeps routing fully
> transparent. SQL is still generated by the model in the `query_orders` tool, then validated.

## 2. Tech stack & why

| Layer | Choice | Reasoning |
|------|--------|-----------|
| **LLM** | **Hugging Face** router → `meta-llama/Llama-3.3-70B-Instruct` (open-source) | OpenAI-compatible endpoint, strong at JSON routing and text-to-SQL — no vendor lock-in. The client is provider-agnostic: swap `OPENAI_BASE_URL`/`CHAT_MODEL` to use OpenAI or Gemini instead. |
| **Embeddings** | **local** `BAAI/bge-small-en-v1.5` (384-d) via `fastembed` (ONNX, CPU) | Open-source and runs **on-box with no API call**, so embeddings can never be rate-limited or 503 — the failure mode that plagued the hosted Gemini path. Tiny corpus → negligible CPU cost; pairs with pgvector cosine. |
| **Backend** | FastAPI + Uvicorn | Async + first-class **SSE token streaming** (`sse-starlette`). |
| **Agent** | JSON-routing loop (no LangChain, no native tool-calling) | Transparent and **defensible** routing that works across any provider; trivial to surface *which* tool ran. Fewer moving parts than a framework. |
| **Data store** | PostgreSQL 16 + **pgvector** | **One** dependency holds *both* the orders table (SQL) and the document vectors (RAG). Production-like, deploys cleanly. |
| **PDF parsing** | `pypdf` | Lightweight, deterministic text extraction. |
| **Chunking** | custom section-aware splitter | These policies are headed (`1. Return Window`); chunking on headings yields meaningful **citations**. |
| **SQL safety** | `sqlparse` + guard + read-only txn | Single-SELECT only, forbidden-keyword block, auto-LIMIT, statement timeout. |
| **Frontend** | Next.js 14 (App Router) + Tailwind | Streaming chat; shows tool badge, citations, and generated SQL/rows. |
| **Packaging** | Docker + docker-compose | `docker compose up` runs DB + API + Web. |
| **Deploy** | Render (`render.yaml`) | Free managed Postgres (pgvector supported) + two web services. |

## 3. Dataset — what it is and how each file is used

All files come from the provided Google Drive folder and live under [`data/`](data/).

### Unstructured knowledge → `data/documents/*.pdf` (used by **RAG only**)

| File | Used for | Key facts |
|------|----------|-----------|
| `returns_policy.pdf` | refund/return questions | **30-day** return window, eligibility, refunds in 7–10 business days |
| `warranty_policy.pdf` | warranty questions | **12-month** warranty (6-month accessories), coverage, claims |
| `product_faq.pdf` | shipping/status/payment/cancellation | order-status meanings, 3–5 day delivery, payment methods |
| `pricing_discounts_policy.pdf` | pricing/discount questions | 10% bulk discount (10+ units), promo-code rules, no price match |
| `hr_leave_policy.pdf` | HR questions | 18 annual / 12 sick leave days (also a great **routing negative test**) |

**Pipeline:** `pypdf` → section-aware chunking (27 chunks total) → local `bge-small-en-v1.5` (`fastembed`)
→ stored in `documents(source, section, ordinal, content, embedding)`. Retrieval is cosine
similarity (`embedding <=> query`), top-`k=4`, returning `source` + `section` as the citation.

### Structured knowledge → `data/orders.csv` (used by **text-to-SQL only**, never embedded)

200 rows, 0 nulls, 0 duplicate IDs. Loaded verbatim into the `orders` table:

| Column | Type | Notes |
|--------|------|-------|
| `order_id` | TEXT | format **`ORD-1001`** (string, not numeric — the SQL prompt is told this) |
| `customer` | TEXT | 15 distinct names |
| `product` | TEXT | 10 distinct products |
| `amount` | INTEGER | whole INR, ₹1,299–₹26,997 |
| `status` | TEXT | `pending` `processing` `shipped` `delivered` `cancelled` `returned` |
| `order_date` | DATE | 2025-12-17 → 2026-06-14 |

**Ground-truth cross-checks (today = 15 Jun 2026):** pending orders = **24**;
"last month" (May 2026) revenue = **₹2,88,332** across **38** orders. Use these to grade answers.

> **Why this split matters:** the `status` vocabulary in the CSV is *defined* by
> `product_faq.pdf` / `returns_policy.pdf`. That's what makes **mixed questions** real —
> e.g. _"Our policy allows 30-day returns; did order ORD-1207 qualify?"_ needs the policy
> (RAG) **and** that order's date/status (SQL).

## 4. How routing works

Routing is an **LLM JSON decision**, not brittle keyword rules. The router prompt
([`backend/app/agent/prompts.py`](backend/app/agent/prompts.py)) describes each tool's
purpose and asks the model to return `{search_query, orders_question, direct_answer}`;
[`orchestrator.py`](backend/app/agent/orchestrator.py) parses it and runs the chosen tools:

| Question type | Example | Router output → tool(s) |
|---|---|---|
| Document | "What is the refund window?" | `search_query` → `search_documents` |
| Data | "How many orders are pending?" | `orders_question` → `query_orders` |
| Mixed | "Policy allows 30-day returns; did ORD-1207 qualify?" | **both** fields → both tools |
| Out-of-scope | "What's the capital of France?" | `direct_answer` → `"I don't have that information."` |

Grounding is enforced in the prompt: cite documents inline (`[returns_policy.pdf]`),
report SQL numbers exactly, and never invent policy text, rows, or columns.

---

## 5. Text-to-SQL safety (defence in depth)

Generated SQL passes through [`app/sql/guard.py`](backend/app/sql/guard.py) before it runs:
1. Strip markdown fences / trailing `;`.
2. `sqlparse` → require **exactly one** statement.
3. Must start with `SELECT`/`WITH`; **forbidden-keyword** block (`INSERT/UPDATE/DELETE/DROP/…`).
4. Auto-append `LIMIT` when missing.
5. Execution runs in a **READ ONLY** transaction with a **statement timeout** — so even a
   guard bypass cannot mutate data. The schema prompt pins the `ORD-` id format, the exact
   `status` values, and the fixed business date, preventing hallucinated columns / bad dates.

## 7. Run locally (Docker — recommended)

```bash
cp .env         
docker compose up --build
```

> The chat LLM uses a **Hugging Face access token** — create a free "Read" token at
> https://huggingface.co/settings/tokens and paste it into `OPENAI_API_KEY`.
> Embeddings run **locally** (`fastembed`), so they need no key and never rate-limit.
> To use OpenAI/Gemini instead, see the commented block in `.env.example`.
- UI → http://localhost:3000
- API health → http://localhost:8000/health  (shows `orders` and `doc_chunks` counts)

On first boot the API waits for Postgres, then **ingests** (loads 200 orders, embeds 27
chunks) automatically. Re-ingest anytime with `make ingest` (or `FORCE_INGEST=1`).
